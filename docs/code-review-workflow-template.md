# 代码审查工作流（通用方法论模板）

> **用途**：这是跨项目通用的代码审查方法论框架，所有项目适用。
> **与项目文档的关系**：本文件是**骨架**；每个项目应在 `docs/代码审查工作流升级方案.md` 提供实例化（填入该项目的扫描器、命令、框架特有鉴权模式、例外规则）。审查时优先读项目实例化文档，无则降级用本模板并提示用户实例化。
> **适用场景**：vibe coding（AI 生成代码比例高），需要对抗 AI 自审自的盲区。

---

## 1. 设计原则

1. **对抗 > 数量**：每条发现经独立 skeptic 反驳，过滤"AI 认可自己的判断"。
2. **验证是审查的一部分**：修复必须追调用链 + 跑回归，不是审完就完。
3. **结构化维度 > 自由发挥**：盲区变成必答 checklist，不能用统计结论代替逐点核对。
4. **根因优先于症状（第一性原理）**：补丁治症状、根因治复发。
5. **攻击者视角**：skeptic 不只判"是不是 bug"，更问"恶意攻击者会怎么利用"。
6. **代码审查 + 基础设施审查 分离**：部署盲区用独立 checklist。
7. **审查-修复-再审查闭环**：不通过必须修复后重审，不允许"我改完了"代替验证。

> 协同闭环：**生成 → 验证 → 本质（第一性原理）→ 修复 → 再验证 → 通过**。

---

## 2. 触发流程（框架，具体命令见项目实例化文档）

当用户说"代码审查 / review / CR"等触发词时，按顺序执行：

| 步骤 | 阶段 | 说明 |
|---|---|---|
| 0 | **阶段 A：确定性扫描打底** | 项目扫描器输出 JSON finding（零模型、零幻觉、可复现） |
| 1 | **阶段 B：对抗式审查** | 审查 agent + skeptic（攻击者 lens + 四态）+ 第一性原理根因升华；无 actionable finding 自动跳过 |
| 2 | `/code-review` | 多代理审查（如可用） |
| 3 | `/simplify` | 过度复杂 / 冗余 |
| 4 | `opengrep` | 安全 / SAST 扫描（免登录） |
| 5 | 验证闭环 | 跑测试验证（按级别分层） |
| 6 | 汇总 | 给出「通过 / 不通过」 |
| 7 | 若「不通过」 | 修复 P0/P1 + 最小验证，**重新触发流程** |

**循环终止条件**：无 P0/P1 + 无未解决 `plausible_blocking` + **验证通过**，方可「通过」。

---

## 3. 按风险分层触发

多代理对抗耗 token、慢，按改动风险分层：

| 风险层 | 触发条件 | 审查方式 |
|---|---|---|
| **高危** | 碰鉴权 / 数据库 / 计费 / 部署 / 跨存储删除 / 配置文件 | 阶段 A+B+C 全跑 |
| **中危** | 改业务逻辑 / 状态机 / 外部服务调用 | 阶段 A + 相关 2-3 维度 |
| **低危** | 纯 UI 样式 / 注释 / 小重构 / 文档 | 阶段 A 确定性扫描 + opengrep |

---

## 4. 分阶段执行

- **阶段 A（确定性扫描）**：事实清单（路由 / 鉴权 / SSRF / 硬编码 / 跨存储删除 / 裸 fetch / 错误吞没 / OOM / 时间戳污染）。输出事实，非判断。
- **阶段 B（审查 + 对抗 + 根因）**：逐条 pipeline，每条 finding 独立跑 review → skeptic → rootcause。
- **阶段 C（多 finder 并行）**：维度并行 + 多 skeptic 对抗（阶段 B 稳定后再上）。

---

## 5. 四态状态机

| 状态 | 含义 | 处置 |
|---|---|---|
| `confirmed` | 已读码或复现确认 | 进入修复 + 验证闭环 |
| `plausible_blocking` | 证据不足但影响高，不能放行 | 补证据后才能降级/确认；**不随低分丢弃** |
| `needs_repro` | 需运行验证才能定性 | 进验证闭环 |
| `refuted` | 明确误报 | 丢弃，记录进漏报台账复核 |

**转换规则**：skeptic 默认从 `plausible_blocking` 起评；只有正面反驳证据才转 `refuted`；`plausible_blocking` 不得因"证据不足"降级。

---

## 6. 维度 checklist（通用）

**鉴权 5 问**（每个写接口必答）：
1. 有鉴权依赖？
2. user_id 从服务端 token 解析，不信任请求体 / query / form？
3. 资源归属比对？
4. 批量操作：每一项都校验？
5. 横向越权（IDOR）：能否用别人的 ID 访问 / 修改？

**跨存储清理**：删任何实体 → 列全部下游存储 → 逐个确认删除顺序 + 失败回滚。

**SSRF**：接受外部 URL 的接口是否做 SSRF 校验？

**修复副作用预判**：每个被改动的 key/函数/字段 → grep 全部消费方 → 列影响面。

**攻击 / 故障模式**：OOM 死循环、时间戳污染、部署元数据缺陷、资源未释放、错误吞没（`except: pass`）。

> **【关键】框架例外规则**：不同框架有"by design 不需要鉴权"的端点（如登录/注册/验证码/refresh/logout），以及"无法加标准鉴权"的端点（如 img 标签路由、sendBeacon）。这些**例外规则必须项目实例化**——见 §9。

---

## 7. 第一性原理（根因升华 + 修复检查）

**根因升华 4 问**（对每条 confirmed/plausible_blocking）：
1. 本质是什么（剥离表象）？
2. 为什么出现（底层原因，非"开发者忘了"）？
3. 同类还有哪些（同一根因的其他实例）？
4. 是孤立失误（isolated）还是系统性设计缺陷（systemic）？

**第一性原理修复检查 4 问**（修复前）：
1. 治症状还是根因？
2. 隐含假设成立吗？
3. 覆盖所有同类情况吗？
4. 不治根因会在哪复发？

---

## 8. skeptic 三 lens

- **正确性**：代码逻辑是否真的错？
- **可复现**：能否构造触发路径？
- **攻击者视角**：恶意用户会怎么利用？（把"逻辑没坏但可被滥用"从"不是 bug"翻成"可被攻击"）

---

## 9. 项目实例化清单（新项目必须填写）

新项目接入本工作流时，在 `docs/代码审查工作流升级方案.md` 至少填清以下：

| 项 | 说明 |
|---|---|
| 确定性扫描器 | 项目扫描脚本名 + 输出 JSON 路径（参考 qzwx 的 `review_scan.py`） |
| 对抗审查 workflow | 阶段 B 的 workflow 文件名 |
| 框架鉴权模式 | 该项目鉴权依赖怎么写（如 FastAPI `Depends`、Spring `@PreAuthorize`、filter） |
| **例外规则** | by design 不需鉴权的端点 + 无法加标准鉴权的端点（img/sendBeacon 等）及**替代控制** |
| 子系统路由 | 多子系统工作区的目录结构 + git worktree 关系（避免误审/混审） |
| 分层触发关键词 | 该项目判定高危/中危/低危的路径关键词 |
| finding schema | 符合通用 schema 的 category 枚举（含框架特有维度） |
| 漏报台账 | `docs/review-miss-ledger.md` 路径 |

> **例外规则是实例化的核心**：阶段 B 验证证明，例外规则不能只写"不加 JWT"，必须补"替代控制"——否则无鉴权+可枚举=IDOR（如 sendBeacon 可被匿名覆盖任意资源）。

---

## 10. 验证闭环（分层验证）

修复后必须跑验证，不能只靠重扫。

| 级别 | 触发条件 | 验证要求 |
|---|---|---|
| P0 / 高危 | 任意 P0，或改动鉴权/数据库/计费/部署/跨存储删除 | 跑项目完整测试套件 + 复现红→绿 + 追调用链声明副作用 |
| P1（鉴权/计费/DB写/外部付费API） | 涉及上述领域 | 跑相关模块测试或最小验证脚本（至少一个正向 + 一个越权/边界负向用例） |
| P1（其他） | 其他 P1 | 最小验证或构建/类型验证 |
| P2 / 低危 | 纯 UI/注释/文档/小重构 | 构建/类型验证 |

验证失败 = 不通过，必须修复后重新触发流程。
未配置测试的项目必须补齐至少构建/类型检查，否则不得通过。

验证命令由项目 `docs/代码审查工作流升级方案.md` 或 `.code-review.yml` 指定。通用扫描器支持 `review-scan --verify` 自动发现和执行。
